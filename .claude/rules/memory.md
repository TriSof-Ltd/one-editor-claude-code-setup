# Memory Rules

## What to save to memory

After learning something important about this project, save it to auto memory. Key things to remember:

- **What the app is** — the user's description of what they're building, updated as it evolves
- **Architecture decisions** — "we use X pattern for Y", "don't use Z library"
- **User preferences** — coding style, design preferences, specific requests they've repeated
- **What's been built** — major features completed, so you don't rebuild them
- **What didn't work** — approaches that failed, so you don't repeat them
- **Key domain knowledge** — business logic, terminology, data relationships

## When to update memory

- When the user describes or re-describes what the app should be
- When the user corrects your approach ("no, not like that, do it this way")
- When you discover something about the codebase that would be useful next session
- When a feature is completed successfully
- When the user changes direction ("actually, let's make it X instead of Y")

## The project description evolves

The initial description in PLAN.md is a starting point. As the user refines their vision through conversation, update your memory with the latest understanding. The memory is the living truth about what this project is — PLAN.md is the structured roadmap.
